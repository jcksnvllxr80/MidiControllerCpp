# Microcontroller target (Pico 2 W / RP2350)

Phase 3 adapters that run the same domain core on hardware. The desktop sim is
unchanged; only the adapters and the build system differ.

## Build & flash

Build the `.uf2` in WSL (same shell you run `make` in), then flash it to the
board. Because the repo lives on `C:\`, the built `.uf2` is also a normal Windows
file you can drag-and-drop.

### 1. One-time prerequisites (WSL / Ubuntu)

```sh
sudo apt update
sudo apt install -y cmake gcc-arm-none-eabi libnewlib-arm-none-eabi build-essential python3 git
# Pico SDK with pico2_w (RP2350) support — needs >= 2.1.0:
git clone --branch master --recurse-submodules https://github.com/raspberrypi/pico-sdk ~/pico-sdk
echo 'export PICO_SDK_PATH=$HOME/pico-sdk' >> ~/.bashrc && source ~/.bashrc
```

### 2. Build

```sh
cd /mnt/c/Users/A-A-Ron/git/MidiControllerCpp/MidiControllerCpp
cmake -B build-pico -DPICO_BOARD=pico2_w
cmake --build build-pico -j        # -> build-pico/midicontroller_pico.uf2
```
Or just **`make build`** from the repo root — it wraps the two commands above and
defaults `PICO_SDK_PATH` to `$HOME/pico-sdk` (override: `make build PICO_SDK_PATH=...`).
(`make build-sim` builds the desktop simulator instead; `make pico-clean` wipes `build-pico/`.)
The editor link (and debug output) run over the USB CDC automatically — no flag
needed. WiFi (CYW43 + lwIP + mDNS) is built in too (the board is `pico2_w`).
CMake runs `tools/embed_data.py` (needs Python 3) to bake `data/*.json` into flash;
exceptions are enabled (`PICO_CXX_ENABLE_EXCEPTIONS`) because the loader throws on bad
config — and `Application::setup()` catches that and falls back to a safe, still-reachable
shell so a corrupt config can't brick boot.

### 3. Flash (BOOTSEL drag-and-drop — easiest on Windows)

1. Hold the **BOOTSEL** button while plugging the Pico into USB. It mounts as a
   drive named **RP2350**.
2. Drag `...\MidiControllerCpp\build-pico\midicontroller_pico.uf2` onto that drive
   in Explorer (or `cp build-pico/midicontroller_pico.uf2 /mnt/e/` to the drive
   letter it mounted as). It reboots and runs immediately.

**Alternative — picotool** (run on Windows; WSL needs `usbipd` USB pass-through):
```sh
picotool load -x build-pico/midicontroller_pico.uf2   # -x = run after loading
```

### 4. Connect / debug

The device enumerates as a USB serial port named **"MidiController"**. Both the
editor protocol and debug `printf` share that one CDC (the app's codec skips
non-JSON noise). Open the COM port at 115200 to watch traffic or talk to it
directly. The editor app connects via its **Serial** transport (not the USB-raw
one, which is a stub for CDC devices).

> Wire your hardware to match `include/mc/adapters/mcu/Pins.h`. See
> [`wiring.md`](wiring.md) for the full build — pin map, external parts, power, and a
> diagram ([`wiring.excalidraw`](wiring.excalidraw)). The build compiles clean and the
> **USB editor link is confirmed working on hardware** (over serial). **WiFi is
> compile-verified only** — not yet tested on a real network.

## Adapters (`include|src/adapters/mcu/`)

| Adapter | Port | Notes |
|---|---|---|
| `McuClock` | IClock | `time_us_64()` |
| `McuMidiOut` / `TeeMidiOut` | IMidiOut | UART @31250 8N1, one per DIN jack; Tee mirrors to both |
| `McuTempoOut` | ITempoOut | square wave at beat rate on 4 GPIOs via a repeating timer |
| `McuLed` | ILed | 3-channel PWM; colour names match RgbKnob |
| `McuInput` | IInput | debounced footswitches (reuses domain `ButtonSM`), quadrature encoder, rotary push-button hold timing |
| `Ssd1306Display` | IDisplay | 128x64 SSD1306 over I2C; splits on " - " into rows |
| `McuConfigStore` | IConfigStore | reads config baked into flash; writes go to a RAM overlay, persisted via `FlashKv` |
| `FlashKv` | (persistence) | stores the write overlay in one reserved flash sector (`KvCodec`, host-tested) |
| `EditorProtocol` (app) | — | newline-JSON request/response; shared by the USB and WiFi links |
| `StdioConfigTransport` | IConfigTransport | runs `EditorProtocol` over the USB CDC |
| `WifiManager` | IWifi | CYW43 STA join, lwIP TCP server (same protocol) on `:8080`, mDNS, creds in flash |

Pin assignments live in one place: `include/mc/adapters/mcu/Pins.h` — edit to match your
wiring. The full build (pin map, external parts, power, diagram) is in [`wiring.md`](wiring.md).

## Status of the hardware-only pieces

These can't be compiled/flashed without the SDK + board, so they're marked by how
far they're verified:

- **OLED font** — DONE. Full ASCII 0x20–0x7E in `Ssd1306Font.h`, generated by
  `tools/gen_font.py` from the public-domain Adafruit GLCD font.
- **USB editor link** — DONE and **confirmed working on hardware** (over serial):
  `identify`/`ping`/`list_*`/`get_*`/`write_*`/`dpad`/`short`, plus `delete_set`/
  `delete_song`/`delete_pedal`/`delete_part` and `write_part` (via the new
  `IConfigStore::remove`; the flash store tombstones build-time-embedded entries
  since their blobs can't be erased). `write_part`/`delete_part` read-modify-write
  the song file (parts have no file of their own). Also `reboot_bootloader` (drops
  into USB BOOTSEL via `reset_usb_boot` — flash a `.uf2` with no front button) and
  `reboot`, both behind `ISystemControl`/`McuSystemControl` (the reset is deferred a
  few ms so the ack flushes first). `identify` now also returns a `device_id` (the
  RP2350 board id) so the editor can dedupe one unit seen over both USB and WiFi.
  A second **raw-USB** path was added (`-DMC_ENABLE_USB_EDITOR=ON`): a vendor-class
  interface (`usb/usb_descriptors.c` + `UsbConfigTransport`, bulk IN/OUT) with
  MS-OS-2.0 descriptors so Windows auto-binds **WinUSB**, for the host app's `nusb`
  transport. **Unverified on hardware** (no SDK/board; `PICO_SDK_PATH` unset in WSL).
  Must match host `transport/usb.rs`: VID 0xCAFE/PID 0x4001, iface 0, EP OUT 0x01 /
  IN 0x81; stdio moves to UART in that build.
- **Flash persistence** — DONE (codec host-tested) and in use: WiFi creds *and* the
  "save defaults" (current set/song/part, written back debounced on change) persist
  via `FlashKv`. Erase/program now goes through `flash_safe_execute()` (XIP-safe;
  `PICO_FLASH_ASSUME_CORE1_SAFE=1` since core1 is never launched — falls back to an
  IRQ-masked write if the safe path is unavailable). Still one reserved 4 KB sector;
  large multi-file edits are a multi-sector follow-up.
- **Reliability** — DONE: an 8 s hardware **watchdog** (enabled after boot init, fed
  each main-loop pass) reboots on a hang; `Application::setup()` falls back to a safe
  shell on bad config; the WiFi connect/reconnect is non-blocking so it can't stall
  the loop under the watchdog; "save defaults" is debounced (1.5 s idle) so a part
  change mid-song doesn't fire an IRQ-masked flash write.
- **WiFi** — built and **compile-verified only (not network-tested)**. First-time
  setup is over USB (`wifi_set`); on boot it auto-joins the saved network
  (non-blocking) and **auto-reconnects** if the link drops, serves the same protocol
  over TCP `:8080`, and advertises via mDNS (`midicontroller.local` /
  `_midicontroller._tcp`). On/off via `wifi_enable`. The app side (a WiFi transport +
  "Set WiFi" UI) is specced in [`wifi-app-handoff.md`](wifi-app-handoff.md).
- **Tap tempo** input isn't wired (no rig pedal uses it); `ITempoOut` is ready.
- **On-target RAM check** — confirm the peak by parsing `BigSky.json` on device.

## Why RP2350 / Pico 2 W

520 KB RAM (vs RP2040's 264 KB) is what lets the build keep nlohmann/json
uncompromised: config is parsed one file at a time, peak is the largest pedal's
JSON DOM (~hundreds of KB worst case) which the 2 W absorbs comfortably. PIO,
dual M33 + FPU, and native USB cover MIDI timing, tempo pulses, and the host link.
Confirm the real RAM peak by parsing BigSky on-target once.
